08 ELF和静态链接

编译、链接和装载

c语言可以通过编译器编译成汇编代码，然后汇编代码再通过汇编器变成 CPU 可以理解的机器码，于是 CPU 就可以执行这些机器码了

如果将之前得add函数拆分成add_lib.c和link_example.c再分别编译

会发现link_example.o无法执行，此时生成的link_example.o并不是可执行文件（Executable Program）而是目标文件（Object File）因为此时两个程序之间还没有联系，用call调用add函数时，计算机并不知道下一个地址，只有通过连接器(Linker)将多个目标文件以及调用的各种函数库链接起来，我们才能得到一个可执行文件。

于是乎，“C 语言代码 - 汇编代码 - 机器码” 这个过程，再计算机上进行时由两部分组成

第一个部分由编译（Compile）、汇编（Assemble）以及链接（Link）三个阶段组成。生成了一个可执行文件。

第二部分，我们通过装载器（Loader）把可执行文件装载（Load）到内存中。CPU 从内存中读取指令和数据，开始真正执行程序。

ELF格式和链接

即可执行文件的格式——可执行文件的内容

ELF 文件格式把各种信息，分成一个一个的 Section 保存起来。ELF 有一个基本的文件头（File Header），用来表示这个文件的基本属性，比如是否是可执行文件，对应的 CPU、操作系统等等。除了这些基本属性之外，大部分程序还有这么一些 Section：

  1. 首先是.text Section，也叫作代码段或者指令段（Code Section），用来保存程序的代码和指令；
  2. 接着是.data Section，也叫作数据段（Data Section），用来保存程序里面设置好的初始化数据信息；
  3. 然后就是.rel.text Secion，叫作重定位表（Relocation Table）。重定位表里，保留的是当前的文件里面，哪些跳转地址其实是我们不知道的。比如上面的 link_example.o 里面，我们在 main 函数里面调用了 add 和 printf 这两个函数，但是在链接发生之前，我们并不知道该跳转到哪里，这些信息就会存储在重定位表里；
  4. 最后是.symtab Section，叫作符号表（Symbol Table）。符号表保留了我们所说的当前文件里面定义的函数名称和对应地址的地址簿。

链接器会扫描所有输入的目标文件，然后把所有符号表里的信息收集起来，构成一个全局的符号表然后再根据重定位表，把所有不确定要跳转地址的代码，根据符号表里面存储的地址，进行一次修正。最后，把所有的目标文件的对应段进行一次合并，变成了最终的可执行代码。

  • 为什么同样一个程序，在 Linux 下可以执行而在 Windows 下不能执行了。其中一个非常重要的原因就是，两个操作系统下可执行文件的格式不一样。
  • 很多程序还依赖各种操作系统本身提供的动态链接库，系统调用等等。需要wine提供对应的实现，兼容格式只是万里长征第一步。

09 程序装载

  • 运行可执行文件时，通过一个装载器，解析ELF或PE格式的可执行文件，将对应的指令和数据加载到内存里来，让cpu执行

    加载后占用的内存是连续的

    同时加载很多个程序，不能让程序自己规定在内存的位置

    那就是我们可以在内存里面，找到一段连续的内存空间，然后分配给装载的程序，然后把这段连续的内存空间地址，和整个程序指令里指定的内存地址做一个映射。

  • 虚拟内存地址/物理内存地址

内存分段

  • 找出一段连续的物理内存和虚拟内存地址进行映射
  • 内存碎片问题-》内存交换

内存分页

  • 减少内存碎片的出现

  • 和分段这样分配一整段连续的空间给到程序相比，分页是把整个物理内存空间切成一段段固定尺寸的大小

  • 一般设置成4KB

  • 可以在进行虚拟内存和物理内存的页之间的映射之后，并不真的把页加载到物理内存里，而是只在程序运行中，需要用到对应虚拟内存页里面的指令和数据时，再加载到物理内存里面去。这样一来，任何程序都不需要一次性加载完所有指令和数据，只需要加载当前需要用到就行了

  • 加入一个间接层

  • 通过引入虚拟内存、页映射和内存交换，我们的程序本身，就不再需要考虑对应的真实的内存地址、程序加载、内存管理等问题了。任何一个程序，都只需要把内存当成是一块完整而连续的空间来直接使用

10 动态链接

  • “链接”，把不同程序的代码段合到一起，当程序非常多时，内存空间的占用则会较多

Dynamic Link

  • 链接加载到内存中的共享库，被多个程序调用

  • 动态库文件也就是 .dll文件（win) .so(Linux)

  • 共享代码：机器码必须是地址无关的，共享文库编译出来的必须是地址无关码

  • 大部都是地址无关的，而常见的地址相关的代码，比如绝对地址代码（Absolute Code）、利用重定位表的代码等等，都是地址相关的代码。

  • 我们的共享库用的都是同一段物理内存地址，但是在不同的应用程序里，它所在的虚拟内存地址是不同的，我们写的程序就必须明确地知道内部的内存地址分配

  • 动态代码库内部的变量和函数调用都很容易解决，我们只需要使用相对地址（Relative Address）就好了。各种指令中使用到的内存地址，给出的不是一个绝对的地址空间，而是一个相对于当前指令偏移量的内存地址。因为整个共享库是放在一段连续的虚拟内存地址中的，无论装载到哪一段地址，不同指令之间的相对地址都是不变的。

  • @plt 我们需要从PLT——程序链接表找要调用的而函数

  • GOT 全局偏移表 ，在共享库的data section， 数据部分是各个动态链接的应用程序里各加载一份的。

    在GOT种查询当前运行程序的虚拟内存里的对应位置，各程序维护好自己的GOT就可以了，偏移量

13 加法器

  • 异或门与半加器

    基础门电路，输入都是单独2个bit，输出1个bit

    对两个8位bit？

    显然需要一位一位运算

    其中，计算一位数的加法：输入00，01，10，11，输出。。。。

    其中输入00，11输出0，输入01，10输出1，这就是一个·XOR门，异或门就是一个最简单的整数加法所需要使用的基本门电路

    算完个位数，我们需要记录是否发生了进位，以及个位数的结果，所以通过一个异或门计算出个位，一个与门计算是否进位，两者打包，即为半加器

  • 全加器

    半加器可以解决个位的加法问题，但二位就不够用

    需要，加数、被加数、前一位的进位信号三个数相加

    解决：两个半加器和一个或门，合成一个全加器

    第一个半加器，得到是否进位X和对应的两个数相加后的结果Y，再把Y，和上一位位数相加后输出的进位信息U连到第二个半加器上，就又拿到了是否进位的信号U与加和后的结果W

    这个W就是我们在二位上留下的结果，我们把两个半加器的进位输出，作为或门的输入，得到进位信号

14 乘法器

  1. 顺序乘法的实现过程

    就是二进制的竖式计算，由于每位的乘数只能为1或0，实际乘法变为位移和加法

    每一位去乘被乘数，然后位移，最后全部加起来

    最后一步的加法，可以用加法器实现

    例如4位二进制数的乘法，使用输入输出都是4个开关，中间一个开关控制，这是单位的乘法，位移使用错开接线就好了

    但如何去节约晶体管呢？32位整数乘法，不就要32组开关了嘛

    我们使用顺序计算。先用乘数最右侧的个位乘以被乘数，结果写入储存计算结果的开关，然后把被乘数左移一位，乘数右移一位，重复上述过程，把结果累加到原先答案上即可，仅需加法器，左移一位和右移一位的电路即可

    但是，这样N位数字的乘法，就需要N次加法，O(N)，比较慢

  2. 并行加速方法

    也就是多个加法同时进行，类似足球

  3. 电路并行

    门延迟 Gate Delay

    例如，每一个全加器，都要等待上一个全加器的输出，位数越多，越往高位走，等待前面的步骤就越多，即门延迟

    每过一个门电路，就一层延迟，称为T

    一个全加器，3T，4位整数，就是3个加法，3 * 3 = 9

    时钟频率

    如果想要使用更少的电路，结果暂存在寄存器，就需要等待下一个时钟周期的到来

    那如何优化，其实，我们思路被加法器这一封装好的电路蒙蔽了，两个数的加法，当两个数确定时，是否进位也就确定了，其信号可以立即输入到下一加法中，也就是把较少的，多层前后计算依赖关系的门电路，展开成较多的，依赖关系较少的门电路。

    本质上是利用了电路的并行性，电路只要接通，输入的信号自动传播到了所有接通的线路里面，这其实也是硬件和软件最大的不同。

16 浮点数和定点数B

  • 问题1：小数部分的二进制表示？其实和整数部分一样，都表示为2的次方，在十进制转向二进制的时候，是*2，看是否超过1，超过1就记下1，数字减1，从小数点往后运算

  • 这样带来的问题就是，有些小数表示为二进制后，就会出现循环，例如0.1

  • 浮点数的加法：先对齐，再计算

    我们需要把两个的指数位对齐，也就是把指数位都统一成较大的那个，有效数字部分进行位移即可

  • 浮点数的精度丢失

    浮点数在加法的过程中，指数叫小的数，需要进行右移，最右侧的有效位就被丢掉了，显然，相加数的指数位差距越大，造成的损失越大

  • public class FloatPrecision{
    public static void main(String[] args){
        float a = 2000000.0f;
        float b = 1.0f;
        float c = a + b;
        system.out.println("c is " + c);
        float d = c - a;
        system.out.println("d is " + d);
    }
}
//发现，c=a，d=0.0

  • Kahan Summation算法

    我们连续加2000万个1会怎么样？会有不可忽视的损失

    其实这个算法的原理其实并不复杂，就是在每次的计算过程中，都用一次减法，把当前加法计算中损失的精度记录下来，然后在后面的循环中，把这个精度损失放在要加的小数上，再做一次运算。
